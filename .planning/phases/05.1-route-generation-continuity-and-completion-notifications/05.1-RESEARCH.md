# Phase 5.1 Research: Route Generation Continuity and Completion Notifications

## Question

How should WanderLog make AI route generation continue after the user leaves Explore, expose progress globally, restore finished results from history, and notify completion without overbuilding a general notification platform?

## Current Implementation Findings

- Persistence already exists. `lib/db/schema/ai-route.ts` stores user-owned sessions, messages, variants, points, and events.
- `lib/db/queries/ai-route.ts` already supports creating sessions/variants, persisting events/points, marking variants complete/failed, reading one user-owned session, and listing sessions by user.
- `server/api/ai/route.post.ts` currently owns generation and streaming in one request. The server work happens inside a `ReadableStream.start()` tied to the client response.
- `composables/use-ai-route-session.ts` restores one active session id from `sessionStorage`, but `sessionStorage` is tab-scoped and not a durable history browser.
- `components/explore/route-history.vue` is variant history for the current session, not all route-generation sessions.
- Nitro 2.11.12 in `node_modules` includes experimental tasks (`defineTask`, `runTask`) and `event.waitUntil`, but the repo does not currently enable Nitro tasks in `nuxt.config.ts`.
- The repo has no PWA module, service worker, push-subscription schema, or web-push sender. Planning docs explicitly warn not to claim push behavior before those pieces exist.

## Primary Source Notes

- Nitro tasks are documented at `https://nitro.build/guide/tasks` and are still represented as experimental in the installed Nitro types. They can provide a named server task boundary, but runtime durability still depends on the deployment target and process lifecycle.
- MDN Push API docs (`https://developer.mozilla.org/en-US/docs/Web/API/Push_API`) describe the browser-side push flow as service-worker based and subscription driven.
- MDN `ServiceWorkerRegistration.showNotification()` (`https://developer.mozilla.org/en-US/docs/Web/API/ServiceWorkerRegistration/showNotification`) is the browser API for showing notifications through a service worker.
- MDN Notifications API (`https://developer.mozilla.org/en-US/docs/Web/API/Notifications_API`) requires permission and secure-context behavior. In-app notifications should remain the reliable fallback when permission/service-worker/subscription is absent.

## Recommended Architecture

Use a three-layer approach:

1. **Durable route run service**
   - Extract provider-generation loop from `server/api/ai/route.post.ts` into `lib/ai/route-generation-runner.ts`.
   - The runner receives authenticated `userId`, request context, session id, and variant id; it persists events/points exactly once and marks terminal status.
   - Add claim/heartbeat/completion metadata to route variants or a small route-generation job table so stale `generating` work can be detected.
   - Start the runner from a thin POST endpoint using `event.waitUntil(...)` and/or Nitro task `runTask(...)` if tasks are enabled.
   - Keep the current SSE behavior as a progress attachment/replay surface rather than the owner of provider work.

2. **Status and history surfaces**
   - Add list/status endpoints for current user sessions (`/api/ai/route-sessions` or nearby route under existing `server/api/ai/route/`).
   - Return compact summaries for global UI: session id, city, status, active variant, point count, updated time, failure code, and notification state.
   - Add a composable that polls or refreshes status from any page, not only Explore.
   - Add a compact nav/global indicator plus route-session history picker linking back to Explore with a session id query.

3. **Completion notification fallback ladder**
   - Always record completion events in persistent status.
   - Show in-app notification/toast when the app is open and the status composable observes a transition to completed/failed.
   - Only attempt browser push when service worker support, notification permission, VAPID configuration, and a saved user-owned subscription exist.
   - Because this repo currently has no push sender dependency, the first implementation should either:
     - add a small, explicitly approved web-push dependency during execution; or
     - implement subscription/permission UI plus in-app fallback and mark true push delivery blocked until dependency/provider approval.

## Data Model Guidance

Prefer extending existing AI route tables first:

- Add `generationStartedAt`, `generationHeartbeatAt`, `generationCompletedAt`, `notificationStatus`, and optional `failureCode`/`retryCount` if current columns are insufficient.
- If the lifecycle state becomes awkward on `aiRouteVariant`, add a small `aiRouteGenerationJob` table keyed to `sessionId`, `variantId`, and `userId`.
- Push subscriptions, if implemented, should be user-owned and separate from route sessions. Store endpoint, keys, user id, created/updated timestamps, and disabled/failure metadata.

## UX Guidance

- Global indicator belongs in `AppNavBar` or the default layout, because Explore uses `layout: false`; Explore may need its own overlay copy of the same indicator.
- Indicator should be compact: spinner/progress label while generating, success/failure state when terminal, and a link to open the route session.
- History should distinguish route sessions from route variants:
  - sessions: city/request/time/status;
  - variants: refinements inside one session.
- Do not display raw provider JSON or raw prompt/context.

## Risks

| Risk | Mitigation |
|------|------------|
| Client leaves before the server starts the durable run. | Create/persist session and variant before returning success; return ids immediately; only then attach background runner. |
| Serverless/process shutdown interrupts in-process background work. | Persist job state and heartbeat; detect stale `generating` jobs; document runtime constraint; later move to durable queue if deployment requires it. |
| Duplicate route points/events after retry/reconnect. | Use monotonic sequence and idempotent persistence guards or clear ownership of a single active runner per variant. |
| Cross-user status leakage. | Every session/job/subscription endpoint filters by `event.context.user.id`; tests assert ownership filters. |
| Notification overclaim. | In-app notification is required; browser push is gated by actual permission/subscription/sender support and otherwise degrades visibly. |
| New dependency creep. | Do not add web-push dependency unless explicitly approved during execution; keep push delivery behind a narrow adapter. |

## Plan Shape

1. Durable route run service and server lifecycle boundary.
2. Status/history endpoints plus global progress UI and session restore.
3. Notification state, in-app completion notification, and browser push gated adapter/subscription path.
4. Resilience, ownership, and stale-job verification.
