# Phase 5.1 Context: Route Generation Continuity and Completion Notifications

## Purpose

Users can start an AI route generation on Explore and then leave the Explore page without losing the result. The app should keep the generation durable, expose progress from a global interface surface, restore the completed/failed output from route history, and notify the user when generation finishes.

## Current Evidence

- `lib/db/schema/ai-route.ts` already defines user-owned route sessions, messages, variants, points, and events.
- `lib/db/queries/ai-route.ts` already has ownership-safe create, update, restore, and listing helpers.
- `server/api/ai/route.post.ts` persists streamed route events, points, and completion/failure states.
- `server/api/ai/route/[session-id].get.ts` restores a single user-owned session snapshot.
- `composables/use-ai-route-session.ts` stores the active session id in `sessionStorage` and restores it on Explore mount.
- `components/explore/route-history.vue` shows variants for the currently restored session, not a global list of all previous sessions.

## Problem

Current generation is still coupled to the active client streaming request. If the user navigates away, reloads, closes the tab, or loses connection during generation, the browser can cancel the request. Persistence may keep partial progress, but the server is not guaranteed to finish the provider call and save the completed route.

## Scope

- Introduce a durable route-generation job/session state that can continue independently of the visible Explore page.
- Add a user-owned active-generation status endpoint or stream-replay endpoint so any app surface can show progress.
- Add a global UI indicator for active route generation outside the Explore panel, with a link back to the session.
- Add a history/list UI or route-session picker so completed background generations can be found after the user returns.
- Add completion notification behavior:
  - in-app notification when the user is still in the app;
  - browser push only when service worker support, notification permission, and a saved push subscription exist;
  - graceful fallback to in-app/global status when push is unavailable.
- Add sanitized logs/Sentry breadcrumbs for job start, completion, failure, notification delivery, and notification fallback.
- Cover ownership boundaries so users cannot observe or subscribe to another user's route-generation jobs.

## Out of Scope

- Generic feed/travel/event push notifications unrelated to route-generation completion.
- Full offline editing or sync conflict handling.
- Saving generated routes to diary, except where a completion notification links to a route that Phase 6 can later save.
- Adding a broad notification preference center unless a narrow route-generation preference is needed.

## Acceptance Criteria

1. A route generation started from Explore reaches `completed` or `failed` in persistent storage even if the user leaves Explore after the session is created.
2. A global app indicator shows active generation status/progress and does not require Explore to be mounted.
3. Returning to the app lets the user open the completed/failed route session from history.
4. In-app completion notification appears for active app sessions.
5. Browser push completion notification is sent only after explicit permission/subscription setup and never exposes private route details in notification text.
6. Tests cover server job state transitions, restore/history listing, notification fallback behavior, and user ownership checks.

## Planning Notes

- Prefer reusing the existing `aiRouteSession`, `aiRouteVariant`, `aiRouteEvent`, and `findAiRouteSessionsByUserId` model before adding new tables.
- If push subscriptions are needed, add the smallest user-owned subscription table needed for route-completion delivery.
- Do not add a new dependency unless the implementation cannot deliver browser push or durable local development behavior through existing Nuxt/Nitro capabilities.
- If deployment lacks a reliable background worker, document the runtime constraint and implement the smallest safe server-side job model supported by the target environment.
