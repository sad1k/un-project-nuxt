<threat_model>
- Annoying install UX: prompts must be user-dismissable and only appear when browser support says installation is available.
- Unsupported offline claims: UI copy must distinguish install/offline shell availability from unavailable offline route generation, map tiles, provider data, or diary editing.
- Mobile regressions: app shell changes must not create text overflow, nested cards, or overlapping fixed UI on small screens.
- Release readiness drift: final verification must honestly document existing typecheck/build blockers if still present.
</threat_model>

<tasks>
1. Add mobile PWA shell tests first.
   - Create `tests/server/pwa-mobile-shell.test.mjs`.
   - Assert the install prompt composable listens for `beforeinstallprompt`, stores dismissal state, and exposes install/dismiss methods.
   - Assert PWA status UI references service-worker/offline shell state without promising offline generation or editing.
   - Assert `layouts/default.vue` includes the prompt/status components once in the app shell.

2. Add install prompt composable and UI.
   - Create `composables/use-pwa-install-prompt.ts`.
   - Listen for `beforeinstallprompt` on client only.
   - Expose `canInstall`, `install`, `dismiss`, and persisted dismissal state in localStorage.
   - Create `components/app/pwa-install-prompt.vue` with compact, mobile-appropriate UI and no marketing hero styling.

3. Add service-worker/offline status surface.
   - Create `components/app/pwa-status.vue` for subtle update/offline shell status.
   - Use browser APIs defensively so SSR and unsupported browsers degrade silently.
   - Avoid claims about offline diary editing, route generation, map tiles, or AI.

4. Integrate with the app shell and mobile styles.
   - Add the PWA components to `layouts/default.vue`.
   - Add safe-area/mobile shell CSS only where needed in `assets/css/main.css`.
   - Keep layout changes restrained and consistent with the operational app shell.

5. Run final Phase 8 verification.
   - Run all Phase 8 tests plus relevant notification/offline tests.
   - Run `pnpm test:server`.
   - Run `pnpm lint:source`.
   - Run `pnpm typecheck` and `pnpm build`; if existing unrelated blockers remain, record exact summaries.
   - Start the dev server and manually verify `/`, `/explore`, and `/dashboard` at mobile viewport sizes if feasible.
   - Use browser DevTools/Application or Lighthouse PWA checks to verify manifest, service worker, offline fallback, and installability if feasible.

6. Document verification and summarize.
   - Create `08-VERIFICATION.md` with command results, browser/mobile checklist results, and known blockers.
   - Create `08-03-SUMMARY.md` with changed files, verification evidence, and remaining risks.
</tasks>
