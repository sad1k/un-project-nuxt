## Truths

- D-02: The primary profile targets 100 users, 1000 photos, and 1000 posts over 10 minutes.
- D-03/D-17/D-18: The balanced mix includes feed reads, photo uploads, post publishes, custom place creates, and public/globe reads.
- D-08/D-09/D-10: Photo traffic must use signed S3-compatible upload plus app metadata writes, not DB-only fake records.
- D-11/D-13: Mutating traffic must be distributed across many synthetic user owners.
- D-05/D-06/D-07: Every scenario step must feed timing, status, timeout, error, and threshold metrics.
- Full storage upload must be guarded by explicit opt-in such as `LOAD_ENABLE_STORAGE_UPLOAD=1`.

</must_haves>

<tasks>

1. Add failing source tests for the e2e scenario shape.
   - Extend `tests/server/load-runner-source.test.mjs`.
   - Assert the new scenario contains the real flow endpoints: `/api/place-photos/create-private`, signed image upload, image metadata write, image visibility patch, `/api/posts`, `/api/feed`, `/api/public/place-photos`, and `/api/public/feed-globe`.
   - Assert the scenario declares target totals for 1000 photos and 1000 posts.
   - Assert full storage upload is opt-in and dry-run friendly.
   - Assert every mutating step is classified as `write` and every feed/public step as `read`.

2. Add deterministic image fixture generation.
   - Create `tests/load/lib/load-image-fixture.mjs`.
   - Generate a tiny deterministic JPEG or PNG body at runtime; avoid committing large binary fixtures.
   - Compute SHA-256 checksum in the same base64 format used by the client upload flow.
   - Keep fixture size small enough for local repeatability but valid enough for S3 object storage.

3. Add S3-compatible signed upload helper.
   - Create `tests/load/lib/load-s3-upload.mjs`.
   - Request the sign endpoint with authenticated headers and checksum/content length.
   - POST returned fields and generated image bytes as `FormData` to the signed URL.
   - Record upload duration separately from sign and metadata-write duration.
   - Return the key for app metadata insertion and manifest tracking.
   - Fail with a clear message when storage upload opt-in is missing.

4. Implement e2e photo/post user journey.
   - Create `tests/load/lib/load-e2e-social-photo.mjs`.
   - For each selected synthetic user, create a custom place/log through `/api/place-photos/create-private`.
   - Sign and upload the image.
   - Record metadata through the returned image URL.
   - Mark the image public with run-id place metadata and coordinates.
   - Publish the image to the feed through `/api/posts`.
   - Capture created ids and keys into the manifest.
   - Use unique names/captions per run/user/sequence to avoid duplicate location or post conflicts.

5. Schedule the balanced 10-minute profile.
   - Add `e2e-social-photo` scenario to `tests/load/run-load.mjs`.
   - Target 100 users, 1000 photos, 1000 posts, and 10 minutes by default.
   - Preserve the chosen balanced mix: approximately 40% feed reads, 25% photo uploads, 20% post publishes, 10% custom place creates, and 5% public/globe reads.
   - Planner/executor may combine create-private/upload/publish into one journey operation as long as metrics report sub-step timings and final totals still hit 1000 photos/posts.
   - Keep reads running throughout the write window rather than only before or after setup.

6. Add read traffic and status checks.
   - Include `/api/feed?limit=...`, cursor reads where useful, `/api/public/place-photos?limit=...`, bounded public photo reads, and `/api/public/feed-globe?limit=...`.
   - Rotate authenticated and unauthenticated reads as appropriate.
   - Validate response status and minimal response shape without parsing or logging private payloads.

7. Verify and summarize.
   - Run source tests.
   - Run dry-run to confirm resolved totals, duration, opt-in status, and thresholds.
   - If local app/storage are configured, run a very small opt-in smoke profile such as 2 users / 2 photos / 2 posts / 30 seconds before attempting the full profile.
   - Create `13-02-SUMMARY.md` with scenario shape, opt-in behavior, and smoke results or documented blockers.

</tasks>
