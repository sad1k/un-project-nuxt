<must_haves>

## Truths

- D-01: The foundation targets local DB load only; do not design a production load harness.
- D-02/D-11/D-12/D-13: The runner must support 100 synthetic users with distinct seeded sessions and rotate them across virtual users.
- D-04/D-05/D-06/D-07: Metrics must be first-class, per-step, per-class, and threshold-enforceable.
- D-14/D-15/D-16: Load data remains after runs and must be traceable by run id for explicit later cleanup.
- Do not add a new dependency unless implementation proves existing Node APIs and project dependencies cannot support the task.

</must_haves>

<tasks>

1. Add failing source tests for the Phase 13 runner contract.
   - Create or extend `tests/server/load-runner-source.test.mjs`.
   - Assert `tests/load/run-load.mjs` exposes or imports a Phase 13 scenario/foundation instead of embedding untestable one-off logic.
   - Assert metrics code tracks p50, p95, p99, status counts, errors, timeout counts, read/write classes, and threshold failures.
   - Assert the local auth seeding helper writes run-id-marked users and sessions and produces cookie headers without logging full tokens.
   - Assert the preflight references `/api/auth/profile` so seeded sessions are verified through the app.

2. Extract metrics helpers from the current runner.
   - Create `tests/load/lib/load-metrics.mjs`.
   - Preserve current summary behavior while adding per-step p50/p95/p99, timeout counts, read/write class summaries, threshold checks, and JSON-safe output.
   - Classify steps as `read`, `write`, or `setup`.
   - Support separate default thresholds for read p95, write p95, error rate, and timeout rate.

3. Add run id and manifest helpers.
   - Create `tests/load/lib/load-run-manifest.mjs`.
   - Generate a deterministic run id prefix such as `load-YYYYMMDD-HHmmss-random`.
   - Write manifests under a gitignored or clearly documented load-output directory.
   - Store created user ids, safe token suffixes only, created location/log/image/post ids, S3 object keys, report path, and cleanup command.
   - Do not store secrets, raw cookies, full session tokens, CSRF tokens, or provider credentials.

4. Add local DB access helpers for load tooling.
   - Create `tests/load/lib/load-local-db.mjs`.
   - Connect to the same local DB URL shape used by the app, with explicit refusal or warning when the URL does not look local unless an override flag is present.
   - Reuse existing project dependency `@libsql/client` or app DB helpers where safe.
   - Keep helper functions narrow: insert users, insert sessions, find run-marked rows, and delete run-marked rows for later cleanup.

5. Add synthetic auth seeding and preflight.
   - Create `tests/load/lib/load-auth-seed.mjs`.
   - Seed 100 users with run-id-marked names/emails and non-admin role.
   - Seed future-expiring sessions for each user.
   - Build cookie headers compatible with the app's Better Auth middleware.
   - Add a preflight that calls `/api/auth/profile` for seeded cookies and fails before any S3 upload work if auth is rejected.
   - Keep OAuth flows out of scope.

6. Wire the foundation into `tests/load/run-load.mjs`.
   - Add CLI options or subcommands for initializing a Phase 13 local run and printing the run id.
   - Keep existing `smoke`, `public`, `explore`, `auth`, `ai-stream`, and `providers` scenarios backward compatible.
   - Add dry-run output for seeded users, duration, thresholds, and report paths.
   - Ensure auth/session/cookie values are redacted from console output.

7. Verify and summarize.
   - Run focused load-runner source tests.
   - Run `npm run load -- --list` and a dry-run for the new Phase 13 setup path.
   - Create `13-01-SUMMARY.md` with seeded auth design, metrics contract, manifest format, and any auth/CSRF caveats discovered.

</tasks>
