<threat_model>
| ID | Threat | Severity | Mitigation |
|----|--------|----------|------------|
| T-0902-01 | Admin endpoints leak raw request context, prompts, provider responses, or raw event payloads. | high | Serialize explicit allowlisted fields only and add tests forbidding `requestContextJson`, `payloadJson`, provider previews, and auth headers in responses. |
| T-0902-02 | Broad admin query is reachable without admin role. | high | All admin endpoints use `defineAdminHandler` before querying. |
| T-0902-03 | Failure taxonomy is too vague to improve the service. | medium | Add normalized `failureStage`, retryability, and safe explanation mapping. |
| T-0902-04 | User-facing route endpoints accidentally become cross-user. | high | Keep normal `/api/ai/**` endpoints user-scoped and add separate admin query helpers. |
</threat_model>

<tasks>
1. Add admin route-generation tests first.
   - Create `tests/server/admin-route-generation.test.mjs`.
   - Assert admin endpoints live under `server/api/admin/**`.
   - Assert both endpoints use `defineAdminHandler`.
   - Assert normal `/api/ai/**` endpoints still use user-scoped queries.
   - Assert admin list serialization excludes route points and raw payloads.
   - Assert admin detail serialization may include sanitized route points but excludes raw `requestContextJson`, raw `payloadJson`, raw prompts, provider body previews, authorization headers, and secret env names.
   - Assert failure stages include `validation`, `provider`, `parsing`, `persistence`, `diary_save`, `notification`, and `unknown`.

2. Add or derive normalized failure stage.
   - Prefer adding a `failureStage` field to `aiRouteVariant` if filters need direct DB support.
   - If deriving instead, centralize the mapper in `lib/db/queries/ai-route.ts` or a small route diagnostics helper.
   - Map provider errors from `sanitizeProviderError` to `provider`.
   - Map no parseable events / no valid route points to `parsing`.
   - Map diary save failures to `diary_save` where represented in admin summaries.
   - Provide retryability and short safe explanation as derived values.

3. Add admin query helpers.
   - Add list query for route-generation admin summaries with pagination/filter inputs.
   - Support filters for status, failure stage, failure code, provider/model where available, date range, stale state, and maybe retry count.
   - Return operational fields: session id, active variant id, safe user identifier, city, days/interests summary, status, display status, provider/model, point count, timings, retry count, diary status, notification status, failure stage/code, retryability, safe explanation.
   - Add detail query for one session that includes the sanitized route snapshot and safe timeline/events where available.

4. Add admin list/detail endpoints.
   - Create `server/api/admin/route-generations.get.ts`.
   - Create `server/api/admin/route-generations/[session-id].get.ts`.
   - Validate query params and session id with existing H3/Zod-style patterns.
   - Use `defineAdminHandler` before broad reads.
   - Return only explicit allowlisted fields.

5. Verify and summarize.
   - Run `node scripts/run-node-tests.mjs tests/server/admin-auth.test.mjs tests/server/admin-route-generation.test.mjs tests/server/ai-route-status.test.mjs tests/server/release-hardening.test.mjs`.
   - Run `pnpm lint:source`.
   - Create `09-02-SUMMARY.md` with changed files, verification evidence, migration notes, and handoff notes for UI.
