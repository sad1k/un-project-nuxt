<must_haves>

## Truths

- Source order is public WanderLog photos -> Google Places Photos -> optional open/provider fallback -> missing-photo state.
- Public app photos may outrank providers only after safe coordinate/name matching.
- Google media is proxied through the server with fresh references and short-lived caching.
- Missing provider data returns missing slots; no AI/illustrative replacement images.

</must_haves>

<tasks>

1. Add failing tests for source priority and fallback.
   - Create `tests/server/place-media-resolution.test.mjs`.
   - Assert public WanderLog photos are checked before Google provider media.
   - Assert Google provider photos are used when no safe public app photo match exists.
   - Assert missing-photo result is returned when no real source is available.
   - Assert Google photo media paths remain server-only and do not expose API keys to client code.

2. Add public WanderLog photo matching helper.
   - Extend `lib/db/queries/location-log-image.ts` with a focused helper to find public photos near a generated route point.
   - Require public visibility, visible moderation status, public place name/coordinates, and bounded distance/name confidence.
   - Return only safe fields needed for `PlacePhoto`: key/url reference, place label, author/attribution label, coordinates, and confidence.

3. Implement media resolver.
   - Extend `lib/explore/place-media.ts` with `resolveRealPlacePhoto`.
   - First check fresh cache metadata when allowed.
   - Then check public WanderLog photos.
   - Then call Google Places photo resolution using the existing provider adapter or a split photo-specific helper.
   - Add a typed optional adapter hook for Wikimedia/Foursquare but keep it disabled unless config/env exists or implementation stays small.
   - Record cache success/failure metadata through `place-media-cache` helpers.

4. Harden Google photo proxy behavior.
   - Keep `/api/explore/place-photo` authenticated unless a public-safe use case is explicitly introduced.
   - Validate photo resource names strictly.
   - Preserve short-lived cache headers.
   - Avoid S3 persistence for Google media.
   - Normalize 404/unavailable responses without logging raw provider details.

5. Wire place intelligence endpoint.
   - Update `server/api/explore/place-intelligence.get.ts` so generated route place enrichment uses `resolveRealPlacePhoto`.
   - Keep rating/review/cost provider enrichment intact.
   - Ensure `buildPlaceIntelligence` receives a real `PlacePhoto` or null only.

6. Verify and summarize.
   - Run focused media/place tests.
   - Create `11-02-SUMMARY.md` with source order, provider decisions, cache behavior, and known provider configuration gaps.

</tasks>
