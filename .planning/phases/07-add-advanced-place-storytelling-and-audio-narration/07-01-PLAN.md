<threat_model>
- Unsupported facts: story generation must return a typed unavailable state and must not call TTS when factual support is weak.
- Cross-user access: every status, generation, and audio read must validate the authenticated user owns the route session, variant, and route point.
- Credential exposure: OpenAI/TTS and storage credentials must remain server-only and must not appear in `runtimeConfig.public` or client source.
- Raw private logging: do not log prompts, route context, provider headers, generated audio payloads, secrets, or private place history.
- Cost abuse: generation must be explicit, route-point scoped, and idempotent where possible so retries do not create unlimited duplicate audio.
- Requirement drift: `ADVPLACE-02` is partial/deferred; use one default voice only and document that the requirement is not complete in this phase.
</threat_model>

<tasks>
1. Lock expected behavior with focused source tests.
   - Add contract tests for story status values, unavailable reasons, source/status cue metadata, audio metadata, and absence of transcript/story body in the client-facing contract.
   - Add server tests proving weak factual support returns unavailable and does not call the TTS adapter.
   - Add server tests proving route story status/generation/audio access is scoped by authenticated user and route point ownership.
   - Add source tests or assertions proving TTS provider env names stay server-only.

2. Add the route place story contract.
   - Create `lib/explore/place-story.ts` with Zod schemas and TypeScript types for story status, generation request, story response, audio metadata, source support metadata, and failure codes.
   - Include only the title/status/source cues needed by the UI. Do not expose transcript, long story text, or prompt text.
   - Model `ADVPLACE-02` honestly through default-voice metadata and a documented deferred status, not as a completed voice/language-choice feature.

3. Add persistence and ownership-safe queries.
   - Add a `routePlaceStory` table keyed by `userId`, `sessionId`, `variantId`, and `routePointId`.
   - Persist status, failure code, source support summary, selected default voice, audio object key or equivalent storage handle, content type, byte length, duration when known, and timestamps.
   - Add query helpers for lookup, create/update, failed/unavailable transitions, and audio metadata reads that all require authenticated user scope.
   - Register the schema with existing Drizzle exports without modifying unrelated tables.

4. Implement the grounded story provider.
   - Build the audio-guide input from provider place facts and route context/rationale returned through existing place intelligence paths.
   - Enforce a factual support threshold before generation. If provider facts are missing or low confidence, return unavailable with a compact status note.
   - Use native `fetch` for the TTS provider rather than adding an SDK dependency.
   - Use one default voice, preferably configurable through server-only env with a safe fallback.
   - Store generated audio through the existing server storage pattern or an authenticated server-streamed audio path, with sanitized observability.

5. Add authenticated API endpoints.
   - `GET /api/explore/place-story`: return current story availability/status for a user-owned route point.
   - `POST /api/explore/place-story/generate`: on explicit user action, validate ownership, validate support, generate or reuse audio, and return playback metadata.
   - `GET /api/explore/place-story/audio`: stream generated audio only after validating authenticated ownership.
   - Keep responses typed and recoverable: unavailable, failed, generating, available.

6. Verify and summarize.
   - Run the focused story contract/server tests.
   - Run the smallest source lint/typecheck path that covers changed server/data files.
   - If local schema push is needed, run the repo's established local Drizzle/schema workflow and record the result.
   - Create `07-01-SUMMARY.md` with changed files, verification evidence, known blockers, and handoff notes for the UI plan.
</tasks>
