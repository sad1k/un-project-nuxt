<threat_model>
| ID | Threat | Severity | Mitigation |
|----|--------|----------|------------|
| T-1001-01 | Public read endpoint leaks private diary descriptions, private notes, private coordinates, or raw image metadata. | high | Serialize explicit allowlisted public fields only; add source tests forbidding private fields. |
| T-1001-02 | Cross-user publish/unpublish is possible by guessing image ids. | high | Owner mutation endpoint must verify `locationLogImage.userId === event.context.user.id` before updating. |
| T-1001-03 | Photos default to public after upload. | high | Schema defaults and insert path keep visibility private until explicit publish action. |
| T-1001-04 | Hidden/moderated photos remain on public map. | high | Public query filters to public visibility plus visible/approved moderation status only. |
| T-1001-05 | Existing feed `post` semantics accidentally become the public map model. | medium | Keep public photo endpoint/query independent from `post`; use `post` only as reference. |
</threat_model>

<tasks>
1. Add tests for public photo data boundaries first.
   - Create `tests/server/public-place-photos.test.mjs`.
   - Assert `locationLogImage` has private-by-default visibility/public metadata fields.
   - Assert public fields include a public place name, public coordinates, published timestamp, visibility, moderation status, and location accuracy/source metadata or equivalent.
   - Assert public read serialization excludes `description`, private `locationLog.description`, private location/log notes, raw S3 metadata, hidden moderation internals, and non-public images.
   - Assert owner publish/unpublish helpers require `userId`.
   - Assert public read endpoint does not use `defineAuthenticatedHandler`.
   - Assert publish/unpublish endpoint does use `defineAuthenticatedHandler`.

2. Extend the image schema with public visibility metadata.
   - Update `lib/db/schema/location-log-image.ts`.
   - Add visibility enum/string with safe default private.
   - Add snapshot-safe public fields: public place name, public latitude, public longitude, published timestamp, moderation status.
   - Add location accuracy/source fields if useful for UI: accuracy meters and source such as `gps`, `manual`, or `approximate`.
   - Keep diary `description` separate from public display fields.
   - Export narrow TypeScript types/constants for visibility and moderation status.

3. Add owner-scoped public visibility query helpers.
   - Update `lib/db/queries/location-log-image.ts`.
   - Add helper to make an image public for an owner, setting snapshot fields and `publishedAt`.
   - Add helper to make an image private again for an owner without deleting the diary image.
   - Add helper to list public place photos with safe fields and bounding-box or pagination support.
   - Add helper to find public photos for a place/near coordinates if needed by UI.
   - Ensure delete/cascade behavior removes or hides public visibility when the underlying image/log/location is deleted.

4. Add owner mutation endpoint.
   - Create `server/api/locations/[slug]/[id]/images/[image-id]/visibility.patch.ts` or an equivalent kebab-case Nuxt route.
   - Use `defineAuthenticatedHandler`.
   - Validate action `public | private` and snapshot fields with Zod.
   - Validate coordinates and public place name.
   - Return only safe owner-facing image state.
   - Preserve existing delete endpoint behavior and make sure unpublish is non-destructive.

5. Add unauthenticated public read endpoint.
   - Create `server/api/public/place-photos.get.ts` or an equivalent public namespace.
   - Do not require authentication.
   - Validate optional map bounds, limit, cursor, or place query params with Zod.
   - Return only public image key/URL data, public place name, public coordinates, author display name, date, accuracy/source display state, and stable public id.
   - Exclude private, hidden, removed, or unapproved photos.

6. Verify and summarize.
   - Run `node scripts/run-node-tests.mjs tests/server/public-place-photos.test.mjs`.
   - Run `pnpm lint:source`.
   - Record schema rollout steps and any existing type/build blockers in `10-01-SUMMARY.md`.
</tasks>
