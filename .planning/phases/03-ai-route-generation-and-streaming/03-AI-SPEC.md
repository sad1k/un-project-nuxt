# AI-SPEC - Phase 3: AI Route Generation and Streaming

> AI design contract generated by `$gsd-ai-integration-phase`. Consumed by `gsd-planner` and `gsd-eval-auditor`.
> Locks framework selection, implementation guidance, and evaluation strategy before planning begins.

## 1. System Classification

**System Type:** Structured generative pipeline with conversational refinement

**Description:**
WanderLog's Phase 3 AI system turns an authenticated user's explicit Explore context into a sequence of generated route variants. The user submits a city, days, interests, filters, and sidebar-selected personal context; the server streams internal route events that let the Explore map progressively receive route points. Good output is map-ready, user-owned, bounded to selected context, explainable in short place-level rationale, and never exposes raw JSON as the user interface.

**Critical Failure Modes:**
1. The AI route includes fabricated coordinates, impossible day grouping, or route points that Phase 4 cannot render.
2. The system leaks or silently includes unselected saved places, diary logs, provider headers, prompts, secrets, or cross-user data.
3. Streaming exposes raw JSON or a chat-first transcript instead of internal route events plus map-first user feedback.
4. Follow-up refinement overwrites route history instead of preserving route variants for session-level switching.
5. Invalid or partial provider output is persisted as a trusted route variant without schema validation and failure metadata.

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Travel planning / consumer itinerary generation

**User Population:** Authenticated travelers planning a city route from their own interests, saved places, and diary history.

**Stakes Level:** Medium

**Output Consequence:** Generated route variants drive map markers, later map animation, place inspection, and eventual save-to-diary workflows. Bad output can waste user time, reveal private travel history, or make downstream map/save phases unreliable.

### What Domain Experts Evaluate Against

| Dimension | Good (expert accepts) | Bad (expert flags) | Stakes | Source |
|-----------|------------------------|--------------------|--------|--------|
| Geographic plausibility | Places are in or near the selected city and coordinates are valid. | Route includes out-of-city or invalid lat/long points. | Broken map rendering and user mistrust. | Product context + route planner norms |
| Itinerary coherence | Points are grouped by day with plausible pacing and clear timing estimates. | Too many places per day, incoherent ordering, or missing duration estimates. | User cannot act on the route. | WanderLog requirements AIROUTE/MAP |
| Preference fit | Places reflect selected interests, filters, selected saved places, and selected diary context. | Generic tourist list ignores explicit interests or sidebar context. | Core value failure. | Phase 2/3 CONTEXT decisions |
| Explainability | Each point has a short rationale tied to context or interest. | Vague claims like "great place" with no reason. | User cannot trust/compare variants. | Phase 3 CONTEXT D-15 |
| Variant preservation | Follow-ups create traceable route variants and keep the previous route accessible. | Follow-up destroys prior route or loses comparison history. | User loses planning state. | Phase 3 CONTEXT D-01 through D-04 |
| Privacy boundary | Only selected sidebar context is summarized and sent to the provider. | Whole diary/saved history is silently included. | Privacy and cross-user security risk. | Phase 3 CONTEXT D-09 through D-11 |

### Known Failure Modes in This Domain

- Hallucinated places or coordinates: real-looking names with invalid or wrong-city coordinates.
- Overpacked routes: day grouping ignores travel time, opening hours, or fatigue.
- Generic plans: output could fit any city because interests/context were not actually used.
- Context overreach: model receives unselected diary logs or excessive raw descriptions.
- Cost/price fabrication: estimated prices appear authoritative without source/confidence metadata.
- Route variant loss: follow-up edits mutate the only stored result and remove the ability to compare alternatives.

### Regulatory / Compliance Context

- No sector-specific travel-planning regulation identified for this v1 route-generation slice.
- Privacy and data-minimization constraints apply because saved places and diary logs can reveal sensitive travel patterns.
- Provider credentials must remain server-only. Prompt, response, headers, and selected personal context must not be logged verbatim.
- Cost/price fields must be optional and confidence/source-aware because Phase 5 owns richer provider-backed place intelligence.

### Domain Expert Roles for Evaluation

| Role | Responsibility |
|------|---------------|
| Product owner / frequent traveler | Label whether routes feel useful, coherent, and preference-aware. |
| Engineer familiar with map stack | Verify coordinates, day grouping, schema compatibility, and downstream Phase 4 readiness. |
| Privacy/security reviewer | Review context-minimization, user ownership checks, logs, and provider payload boundaries. |

## 2. Framework Decision

**Selected Framework:** Direct OpenAI-compatible Responses API integration via Nitro server endpoint and native `fetch`

**Version:** OpenAI Responses API and Structured Outputs docs current as of 2026-05-10; no new runtime framework dependency required for the first slice.

**Rationale:**
This phase needs one authenticated, stateful, structured route-generation pipeline, not multi-agent orchestration. The app is TypeScript/Nuxt/Nitro, already has Zod, Drizzle, Better Auth, Sentry, and a no-casual-dependencies constraint. Direct Responses API over server-side `fetch` keeps provider credentials server-only, supports SSE-style streaming, avoids adding a framework before the workflow needs one, and lets the team define WanderLog-specific route events and persistence. Official OpenAI docs recommend the Responses API for direct text-generation requests and especially for reasoning models; streaming uses server-sent events, and Structured Outputs supports strict JSON-schema-shaped output for type-safe downstream parsing.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| OpenAI Agents SDK | Helpful for agents, handoffs, guardrails, and tracing, but Phase 3 is a single route pipeline with app-owned persistence. It would add abstraction before agent orchestration exists. |
| LangGraph | Strong for complex stateful graphs and audit trails, but overkill for one linear route-generation/refinement flow in a Nuxt app. |
| LangChain | Broad ecosystem but unnecessary abstraction and dependency surface for direct structured generation. |
| LlamaIndex | Best when retrieval/document ingestion is the core problem. Phase 3 uses bounded selected user context, not a RAG corpus. |
| Native Chat Completions | Older surface for this use case; official docs recommend Responses API for text generation and reasoning models. |

**Vendor Lock-In Accepted:** Partial - use OpenAI-compatible HTTP configuration (`OPENAI_API_KEY`, optional `OPENAI_BASE_URL`, model env var), but do not add provider-management UI in v1.

**Official References:**
- OpenAI streaming responses: https://developers.openai.com/api/docs/guides/streaming-responses
- OpenAI structured outputs: https://developers.openai.com/api/docs/guides/structured-outputs
- OpenAI text generation / Responses API guidance: https://developers.openai.com/api/docs/guides/text
- Latest model guidance checked: https://developers.openai.com/api/docs/guides/latest-model

## 3. Framework Quick Reference

> Fetched from official docs by `gsd-ai-researcher`. Distilled for this specific use case.

### Installation
```bash

# Use native fetch from Nitro/Node and existing zod for validation.
# Optional later dependency, only if explicitly approved:
# pnpm add openai
```

### Core Imports
```ts
import { z } from "zod";

import defineAuthenticatedHandler from "~/utils/define-authenticated-handler";
```

### Entry Point Pattern
```ts
export default defineAuthenticatedHandler(async (event) => {
  const body = await readValidatedBody(event, RouteGenerationRequestSchema.parse);
  const userId = event.context.user.id;

  const selectedContext = await buildSelectedExploreContext(userId, body.context);
  const routeSession = await createRouteSession({
    userId,
    requestContext: body.context,
    followUpToVariantId: body.followUpToVariantId,
  });

  return sendStream(event, async (emit) => {
    emit({ type: "route.session.created", sessionId: routeSession.id });

    const providerStream = await fetchOpenAiCompatibleStream({
      instructions: ROUTE_SYSTEM_INSTRUCTIONS,
      input: buildRoutePrompt(body, selectedContext),
      schema: RouteEventEnvelopeJsonSchema,
    });

    for await (const providerEvent of parseSse(providerStream)) {
      const routeEvent = parseRouteEvent(providerEvent);
      const validated = RouteEventEnvelopeSchema.safeParse(routeEvent);

      if (!validated.success) {
        emit({ type: "route.warning", code: "invalid_event_skipped" });
        continue;
      }

      await persistRouteEvent({ userId, sessionId: routeSession.id, event: validated.data });
      emit(validated.data);
    }

    emit({ type: "route.completed", sessionId: routeSession.id });
  });
});
```

### Key Abstractions

| Concept | What It Is | When You Use It |
|---------|------------|-----------------|
| Route session | User-owned AI planning session containing messages and generated route variants. | Every first route generation and follow-up refinement. |
| Route variant | A generated route candidate inside a session. | When the model creates an initial route or a follow-up changes part of it. |
| Route event envelope | Internal streaming contract with event type, sequence, payload, validation status, and optional provider metadata. | Every progressive update sent from server to client. |
| Route point contract | Schema for map-ready places: coordinates, day, timing, rationale, confidence, alternatives, distance, optional cost. | Persisted in route variants and consumed by Phase 4 map rendering. |
| Context summary | Server-side bounded summary of selected saved places/logs/candidates. | Provider prompt construction; never send whole user history. |

### Common Pitfalls

1. Treating streamed text as the UI. The user clarified the UI should be map-first; route JSON is internal only.
2. Persisting provider output before Zod/schema validation. Invalid output must become warnings or failed variants, not trusted route data.
3. Sending raw diary data or unselected personal context. Sidebar selection is the privacy boundary.
4. Creating only message history and forgetting generated-route variants. Users need to switch variants within a session.
5. Making price mandatory. Cost is optional unless sourced/estimated responsibly with confidence metadata.

### Recommended Project Structure
```text
server/api/ai/route.post.ts              # authenticated streaming route-generation endpoint
server/api/ai/route-sessions.get.ts      # user-owned session/history listing
server/api/ai/route-sessions/[id].get.ts # session + variants detail
lib/ai/route-contract.ts                 # Zod schemas, JSON schema, route event types
lib/ai/route-prompts.ts                  # system instructions and prompt assembly
lib/ai/openai-compatible.ts              # server-only provider fetch/SSE parsing
lib/db/schema/ai-route.ts                # sessions/messages/variants/points/events
lib/db/queries/ai-route.ts               # ownership-safe persistence and reads
tests/server/ai-route-contract.test.mjs  # schema and event contract tests
tests/server/ai-route-security.test.mjs  # ownership/context/logging source checks
```

## 4. Implementation Guidance

**Model Configuration:**
- Use server-only configuration names such as `OPENAI_API_KEY`, `OPENAI_BASE_URL`, and `OPENAI_ROUTE_MODEL`. Add them to `lib/env.ts` only when implementation begins; do not read `.env`.
- Default model should come from env so deployments can tune quality/cost without code changes.
- Start with low temperature (`0.2` to `0.4`) for route contract stability.
- Use strict structured output for final route-event payloads where possible, with local Zod validation as the mandatory enforcement layer.

**Core Pattern:**
1. Validate client request with Zod against the Phase 2 `ExploreRequestContext` boundary.
2. Re-fetch selected saved places/logs server-side by authenticated `userId`; never trust client-provided personal objects.
3. Create a route session and an in-progress variant before provider streaming starts.
4. Stream provider response into internal route events: session created, variant started, point added, point revised, warning, completed, failed.
5. Validate each event envelope before persisting or emitting to client.
6. Persist route messages and route variants; client can switch variants later.

**Tool Use:**
- No LLM tool calling required in Phase 3.
- No web search tool required in Phase 3; city/candidate context comes from Phase 2.
- Optional future tool calls for live place details, weather, reviews, or price sources belong to Phase 5.

**State Management:**
- Persist route sessions, messages, variants, and route points in Drizzle/Turso.
- Every table must carry `userId` or be joined through a user-owned parent in every query.
- Store provider metadata only when sanitized: model id, latency, token counts if available, validation status, error category. Do not store provider headers or full raw prompts by default.
- Store failed variants with status and safe error code if that helps retry UX; do not show stack/provider details to users.

**Context Window Strategy:**
- The provider prompt receives selected sidebar context only.
- Summarize saved places and diary logs into compact records: id, name, coordinates, short description excerpt, date/timing hints, and reason selected.
- Cap counts and string lengths before prompt assembly.
- Keep conversation history as compact variant summaries plus the latest user follow-up, not a growing raw transcript.

## 4b. AI Systems Best Practices

> Written by `gsd-ai-researcher`. Cross-cutting patterns every developer building AI systems needs - independent of framework choice.

### Structured Outputs with Pydantic

The project is TypeScript-first, so implementation should use Zod as the runtime validator. The Pydantic shape below documents the equivalent contract for rubric/eval portability and satisfies the AI-SPEC template's cross-language structured-output requirement.

```python
from pydantic import BaseModel, Field
from typing import Literal, Optional

class Coordinate(BaseModel):
    lat: float = Field(ge=-90, le=90)
    long: float = Field(ge=-180, le=180)

class RoutePoint(BaseModel):
    id: str
    name: str
    day: int = Field(ge=1, le=14)
    coordinates: Coordinate
    estimated_start: Optional[str] = None
    estimated_duration_minutes: Optional[int] = Field(default=None, ge=15, le=720)
    rationale: str
    confidence: Literal["low", "medium", "high"]
    alternative_for_point_id: Optional[str] = None
    approximate_distance_meters: Optional[int] = Field(default=None, ge=0)
    estimated_price_level: Optional[Literal["free", "low", "medium", "high", "unknown"]] = None
    price_confidence: Optional[Literal["low", "medium", "high"]] = None

class RouteEvent(BaseModel):
    type: Literal["route.point.added", "route.point.updated", "route.variant.completed"]
    sequence: int
    variant_id: str
    point: Optional[RoutePoint] = None
```

TypeScript implementation should mirror this with Zod:

```ts
const RoutePointSchema = z.object({
  id: z.string().min(1),
  name: z.string().min(1).max(160),
  day: z.number().int().min(1).max(14),
  coordinates: z.object({
    lat: z.number().min(-90).max(90),
    long: z.number().min(-180).max(180),
  }),
  estimatedStart: z.string().optional(),
  estimatedDurationMinutes: z.number().int().min(15).max(720).optional(),
  rationale: z.string().min(1).max(500),
  confidence: z.enum(["low", "medium", "high"]),
  alternativeForPointId: z.string().optional(),
  approximateDistanceMeters: z.number().int().min(0).optional(),
  estimatedPriceLevel: z.enum(["free", "low", "medium", "high", "unknown"]).optional(),
  priceConfidence: z.enum(["low", "medium", "high"]).optional(),
});
```

### Async-First Design

Use Nitro streaming primitives and async iteration over SSE chunks. Do not block until the whole provider response completes when a valid route point can be emitted earlier. Keep the persistence step in the streaming loop small and idempotent by using `sessionId`, `variantId`, and monotonic `sequence`.

### Prompt Engineering Discipline

- Put non-negotiable behavior in server-owned system instructions: map-first events, no raw JSON UI, selected context only, optional cost with confidence, and no unsupported claims.
- Put user preferences and sidebar context in the request input.
- Include a compact route-event schema description and one or two few-shot examples for adding/revising route points.
- Never let browser-provided text override privacy or ownership rules.

### Context Window Management

- Hard cap selected saved places, diary logs, and candidate places.
- Trim descriptions and diary notes before prompt assembly.
- Use previous route variant summaries for follow-up context instead of full prior provider output.
- If a follow-up targets one point, include only the active route summary plus the target point and relevant alternatives.

### Cost and Latency Budget

- Route generation must stream enough progress to feel alive within the first few seconds.
- Store provider latency and validation result categories for offline review.
- Avoid tool calls in Phase 3 to keep latency predictable.
- Add retry only for transport/provider failures before any route events were persisted; after partial persistence, create a failed variant and let the user retry.

## 5. Evaluation Strategy

### Dimensions

| Dimension | Rubric (Pass/Fail or 1-5) | Measurement Approach | Priority |
|-----------|---------------------------|----------------------|----------|
| Schema validity | Pass if every persisted/emitted route event validates against Zod and required map fields exist. | Code | Critical |
| User ownership | Pass if all reads/writes are scoped by authenticated `userId` and cross-user ids are rejected. | Code | Critical |
| Context minimization | 1-5 score: 5 means only selected sidebar context summarized; 1 means unselected/raw diary data included. | Code + Human | Critical |
| Geographic plausibility | 1-5 score based on city fit, valid coordinates, and no impossible out-of-city points. | Code + Human | High |
| Itinerary coherence | 1-5 score for day grouping, reasonable count per day, timing estimates, and clear order. | Human + optional LLM Judge after calibration | High |
| Preference fit | 1-5 score for alignment with selected interests, filters, saved places, and diary context. | Human + optional LLM Judge after calibration | High |
| Progressive event UX | Pass if route events can be consumed incrementally and raw JSON is not rendered as user-facing UI. | Code + manual browser check in Phase 4 | High |
| Variant preservation | Pass if follow-up generation preserves prior variants and creates/updates traceable variants. | Code | High |

### Eval Tooling

**Primary Tool:** Existing `node:test` server tests + Zod validators for Phase 3; human review checklist for route quality examples.

**Setup:**
```bash
pnpm test:server
```

**CI/CD Integration:**
```bash
pnpm verify:explore-foundation
```

Use Sentry for sanitized provider/runtime failures because it is already installed. Arize Phoenix is a good future AI tracing candidate, but adding it is not required for this phase and should wait for explicit dependency approval.

### Reference Dataset

**Size:** 12 examples to start.

**Composition:**
- 3 happy-path city routes with different durations and interests.
- 2 routes using selected saved places.
- 2 routes using selected diary logs.
- 2 follow-up refinements replacing a route point.
- 1 invalid/ambiguous city or missing city case.
- 1 provider partial/invalid structured output case.
- 1 cost-sensitive route where cost must remain optional or low-confidence.

**Labeling:**
Product owner labels usefulness, preference fit, and itinerary coherence. Engineer labels schema validity and map-readiness. Privacy/security reviewer labels context minimization and user ownership behavior.

## 6. Guardrails

### Online (Real-Time)

| Guardrail | Trigger | Intervention |
|-----------|---------|--------------|
| Auth required | No `event.context.user` | Block with 401. |
| Request schema invalid | Missing city/days/interests or invalid ids | Block with 400/422 and no provider call. |
| Cross-user context id | Selected saved place/log id does not belong to user | Block or omit with audit-safe warning; never send to provider. |
| Provider output invalid | Route event fails Zod/schema validation | Skip event, emit safe warning, mark variant failed if unrecoverable. |
| Raw JSON UI risk | Client receives event type not on allowlist | Drop event and log sanitized client/server warning. |
| Cost unsupported | Price/cost present without source/confidence | Downgrade to unknown or remove before persistence. |

### Offline (Flywheel)

| Metric | Sampling Strategy | Action on Degradation |
|--------|-------------------|-----------------------|
| Invalid route-event rate | All failed validations; weekly sample of warnings | Tighten schema/prompt, add fixtures. |
| Follow-up variant loss | Sample sessions with follow-up requests | Fix persistence/query logic before expanding Phase 4 UI. |
| Context overreach findings | Review payload summaries from test fixtures and sampled safe metadata | Reduce context fields/caps; add regression test. |
| Route quality score | Human review of reference dataset and selected production-like examples | Adjust prompt examples and constraints. |
| Provider latency/failure category | All route-generation attempts as sanitized metrics | Tune model/config, retry policy, or fallback behavior. |

## 7. Production Monitoring

**Tracing Tool:** Sentry for sanitized errors and latency/error breadcrumbs in this phase; Arize Phoenix deferred as an optional AI tracing platform if a future phase approves observability dependency expansion.

**Key Metrics to Track:**
- route generation start/completion/failure counts
- provider latency and timeout categories
- route-event validation failure count
- route variants per session and follow-up success count
- sanitized reason for failed variants

**Alert Thresholds:**
- Provider failure rate above 10% in a rolling hour.
- Route-event validation failures above 5% of generated events.
- Any detected cross-user context access attempt.
- Any server log path attempting to include raw prompt, provider headers, or selected personal context.

**Smart Sampling Strategy:**
- Sample failed variants, retries, sessions with follow-ups, sessions with low-confidence points, and cost-bearing routes.
- Periodically review a small set of successful routes for preference fit and itinerary coherence.
- Prefer metadata and summarized payloads for review; do not export raw diary text or full prompts by default.
