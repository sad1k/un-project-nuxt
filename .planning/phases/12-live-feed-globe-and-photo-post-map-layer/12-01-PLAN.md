## Truths

- D-03/D-08: The globe shows created public photo posts with exact public coordinates, not arbitrary private diary images.
- D-01: This work stays scoped to Phase 12 and should not reopen completed Phase 10/11 implementation decisions except where integration requires it.
- D-04/D-05: Publishing is a quick explicit authenticated action; no automatic feed posting for every public photo.
- D-06: Reuse the existing `post` plus `locationLogImage` model instead of creating a separate globe-post table.
- D-07/D-09/D-15: Public globe reads are unauthenticated and expose only safe public metadata.
- D-14: The public contract must include enough data for popup photo, place, author display name, and date.

</must_haves>

<tasks>

1. Add failing tests for safe public globe payloads.
   - Create `tests/server/feed-globe-public.test.mjs`.
   - Seed or stub a feed post joined to a public `locationLogImage` with `publicLat`, `publicLong`, `publicPlaceName`, visible moderation status, and a public user display label.
   - Assert unauthenticated reads return public globe items with post id, created date, image URL or safe image reference, public place name, exact public coordinates, author display name/avatar if already public, and popup-safe alt text.
   - Assert private image description, diary log title/description, private location notes, raw S3 key, provider internals, route context, email, and auth-only user identifiers are absent.
   - Assert private, hidden-moderation, or coordinate-less images are excluded.

2. Add tests for authenticated quick publish eligibility.
   - Create `tests/server/feed-publish-public-photo.test.mjs`.
   - Assert `/api/posts` accepts a current user's public coordinate-backed image and returns a post shape suitable for the feed.
   - Assert `/api/posts` rejects another user's image, private images, images without public coordinates/place name, and hidden-moderation images.
   - Preserve duplicate image conflict behavior.

3. Add query helpers and serializers.
   - Extend `lib/db/queries/post.ts` with a globe-specific public query such as `getPublicFeedGlobePosts`.
   - Add a focused helper such as `getUserFeedPublishImages` that returns only current-user images eligible or relevant for feed publishing, including `isPosted` and public place metadata.
   - Add or co-locate a serializer that maps database rows to `PublicFeedGlobePost` and strips forbidden fields before endpoint responses.
   - Keep `getFeedWithUserLikes` compatible with existing feed store while optionally adding safe `publicPlaceName`/coordinate metadata for UI use.

4. Harden `/api/posts`.
   - Keep `defineAuthenticatedHandler`.
   - Validate `locationLogImageId` and caption with Zod.
   - Verify the image belongs to the current user and is public, coordinate-backed, place-named, and visible before calling `createPost`.
   - Return clear 403/404/422 errors for owner and eligibility failures.
   - Preserve 409 duplicate behavior.

5. Update publish image source endpoint.
   - Update `server/api/posts/my-images.get.ts` to return the new helper shape.
   - Include enough UI metadata to show which images are ready for quick publishing and which are already posted.
   - Do not include private diary text or raw storage keys beyond existing safe image rendering needs.

6. Add public globe read endpoint.
   - Create `server/api/public/feed-globe.get.ts`.
   - Validate pagination/limit and optional `since` or bounding parameters if useful.
   - Return only serialized public globe posts.
   - Keep it unauthenticated.

7. Verify and summarize.
   - Run focused tests for feed globe public payloads, publish eligibility, existing feed tests if present, and public place photo tests.
   - Run a targeted secret/private-field scan across changed Phase 12 source.
   - Create `12-01-SUMMARY.md` with data contract decisions, endpoint behavior, and verification results.

</tasks>
